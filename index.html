ŌŁÉ TOPIC 1: Secret Key Cryptography 7 Marks
1Introduction to Secret Key
Same key for both encryption and decryption ŌĆö also called symmetric cryptography. Both parties must share the key securely before communication.
2DES (Data Encryption Standard)
56-bit key, 64-bit block size, 16 rounds of operations. Now considered weak ŌĆö 56-bit key is too short and can be brute-forced in hours.
3Triple DES (3DES)
DES applied 3 times: Encrypt(K1) ŌåÆ Decrypt(K2) ŌåÆ Encrypt(K3). Effective key = 168 bits. Much stronger than DES. Used in legacy banking systems.
4AES (Advanced Encryption Standard)
Current gold standard. Key sizes: 128/192/256 bits. Operates on 4├Ś4 byte state matrix with SubBytes, ShiftRows, MixColumns, AddRoundKey operations per round.
5Key Distribution
Securely delivering secret keys. Key Distribution Center (KDC) manages keys centrally. Kerberos protocol is the most popular KDC system in enterprise networks.
6Attacks on Secret Key
Brute Force: Tries all possible keys. Dictionary: Uses precomputed key list. Differential Cryptanalysis: Analyzes plaintext-ciphertext pairs to deduce the key.
7Advantages & Disadvantages
Advantages: Very fast, efficient for large data volumes, simple implementation. Disadvantages: Key distribution problem ŌĆö securely sharing the key without interception is the main challenge.

ŌŁÉ TOPIC 2: Public Key Cryptography 7 Marks
1Introduction to Public Key
Two mathematically related keys: Public key (shared openly) + Private key (kept secret). Message encrypted with public key can ONLY be decrypted with private key.
2RSA Algorithm
Based on difficulty of factoring large prime products (N = p ├Ś q). Used for both encryption AND digital signatures. Most widely deployed public key algorithm worldwide.
3ECC (Elliptic Curve Cryptography)
Based on elliptic curves over finite fields. 256-bit ECC key = security of 3072-bit RSA key. Ideal for mobile devices due to small key size and low power consumption.
4Diffie-Hellman Key Exchange
Two parties generate shared secret over insecure channel WITHOUT sending the actual key. Based on discrete logarithm problem. Used in every HTTPS connection today.
5Java Cryptography Extensions (JCE)
Java APIs for cryptographic operations. Supports AES, RSA, SHA, MD5. Key classes: Cipher (encryption), KeyGenerator, MessageDigest (hashing), KeyPairGenerator.
6Attacks on Public Key
MITM on DH: Intercepts key exchange. Factoring Attack: Factor N to break RSA. Timing Attack: Measures decryption time to deduce the private key value.
7Comparison: Secret Key vs Public Key
Secret key: Fast, 1 key, hard to distribute. Public key: Slow, 2 keys, easy to distribute. In practice: HTTPS uses both ŌĆö RSA/DH to exchange AES key, then AES for all data.

ŌŁÉ TOPIC 1: Hash Functions and MAC 7 Marks
1Hash Function Definition
Takes any-length input ŌåÆ produces fixed-length output (digest). One-way function ŌĆö cannot reverse it. Same input always gives same hash output.
2Properties of Hash Functions
Pre-image Resistance: Can't find input from hash. Collision Resistance: No two inputs share same hash. Avalanche Effect: Tiny change in input = completely different hash.
3MD5 (Message Digest 5)
Produces 128-bit hash. Processes 512-bit blocks through 4 rounds ├Ś 16 operations. Now broken ŌĆö collision attacks found. No longer safe for security applications.
4SHA (Secure Hash Algorithm)
SHA-1: 160 bits (weak). SHA-256: 256 bits. SHA-512: 512 bits. Stronger than MD5. Used in SSL/TLS certificates, digital signatures, and Bitcoin blockchain.
5MAC (Message Authentication Code)
Hash of (message + secret key). Receiver recomputes MAC and compares. Matching MAC confirms BOTH integrity AND authenticity ŌĆö two-in-one check.
6HMAC (Hash-based MAC)
Hash + secret key combined per RFC 2104 standard. Mathematically stronger than plain MAC. Used in SSL, TLS, SSH, and IPSec for authentication.
7Difference: Hash vs MAC
Hash: No key needed, integrity only, can be forged. MAC: Secret key required, integrity + authentication, cannot be forged without the key. Use Hash for: password storage. Use MAC for: API request signing.

ŌŁÉ TOPIC 2: Digital Signatures & Biometric Authentication 7 Marks
1Digital Signature Introduction
Electronic equivalent of handwritten signature. Created by encrypting message hash with private key. Provides Authentication + Integrity + Non-Repudiation simultaneously.
2How Digital Signature Works
Signing: Hash(message) ŌåÆ Encrypt with Private Key ŌåÆ Attach to message. Verification: Decrypt with Public Key ŌåÆ Compare hash ŌåÆ If match = valid, genuine signature.
3RSA Digital Signature
Private key for signing, public key for verification. Based on prime factoring difficulty. Used in SSL certificates, email signing (S/MIME), and software distribution signing.
4DSA (Digital Signature Algorithm)
U.S. NIST Federal standard. Based on discrete logarithm problem. ONLY for signing/verification ŌĆö cannot encrypt data unlike RSA. Faster signing, slower verification than RSA.
5RSA vs DSA Comparison
RSA: Encryption + Signing, fast verify, slow sign. DSA: Sign only, fast sign, slow verify. Both provide strong non-repudiation. RSA more widely deployed globally.
6Non-Repudiation via Digital Signatures
Only sender holds private key ŌåÆ only they could create that signature. Legally binding under IT Act 2000, India. Courts accept as valid legal evidence.
7Biometric Authentication
Physiological: Fingerprint, iris scan, facial recognition, retina scan, hand geometry. Behavioral: Voice recognition, typing rhythm, gait analysis. Cannot be lost, stolen, or forgotten unlike passwords. Used in banking, passports, access control, border control.

ŌŁÉ TOPIC 2: Smart Cards & Zero Knowledge Protocols 7 Marks
1Smart Card Introduction
Portable plastic card with embedded tamper-resistant microprocessor chip. Stores data, executes programs, and performs cryptographic operations securely on the chip itself.
2Application Security via Smart Cards
Private key stored inside chip ŌĆö NEVER leaves the card. All crypto operations (signing, decryption) done on-card. Two-factor authentication: Something you have (card) + Something you know (PIN).
3Zero Knowledge Protocol Definition
Prover proves knowledge of a secret to Verifier WITHOUT revealing the secret. Verifier gains full confidence but learns absolutely nothing about the actual secret value.
4Zero Knowledge ŌĆö Working Steps
1) Prover commits to random value. 2) Verifier sends challenge. 3) Prover responds using challenge + secret. 4) Verifier checks mathematical consistency. 5) Repeat many times until confident.
5Zero Knowledge in Smart Cards
Card proves PIN knowledge without transmitting actual PIN over the network. Prevents eavesdroppers from capturing credentials. Mathematical proof replaces direct PIN transmission.
6Attacks on Smart Cards
Side Channel: Monitor power/EM emissions to extract keys. Fault Injection: Voltage glitches to bypass security. Reverse Engineering: Physically dissect chip under microscope. Replay: Retransmit captured auth messages.
7Protection Against Smart Card Attacks
Tamper-resistant hardware packaging prevents physical access. Encrypted chip storage prevents reverse engineering. Auto-lockout after 3 failed PINs prevents brute force. Power consumption randomization defeats side-channel attacks. Session timestamps prevent replay attacks.
