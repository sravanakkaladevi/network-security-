TOPIC 1 · SECURITY ATTRIBUTES (CIA+) — EVERY ATTRIBUTE EXPLAINED
🔏
Confidentiality
Ensures information is only accessible to those authorised to access it. Prevents unauthorised disclosure.
  • Mechanism: Encryption (AES, RSA), Access Control Lists
  • Threat: Eavesdropping, Sniffing, Shoulder surfing
  • Example: WhatsApp end-to-end encryption keeps messages private
  • Levels: Data confidentiality, Privacy (personal data)
  • Tools: VPN, TLS/SSL, File Encryption, Steganography
✅
Integrity
Guarantees that data has not been altered or tampered with during transmission or storage.
  • Mechanism: Hash functions (SHA-256), Digital Signatures, MAC
  • Threat: Data tampering, Man-in-the-Middle modification
  • Example: SHA-256 checksum verifies downloaded software integrity
  • Types: Data integrity, System integrity, Software integrity
  • Tools: MD5/SHA checksums, Digital certificates, HMAC
🪪
Authenticity
Verifies that the source of information is genuine — the entity is who it claims to be.
  • Mechanism: Passwords, Biometrics, Digital Certificates, PKI
  • Threat: Spoofing, Impersonation, Phishing
  • Example: Certificate Authority verifies website identity via HTTPS
  • Factors: Something you know (password), have (token), are (biometric)
  • Protocols: Kerberos, OAuth, SAML, OpenID Connect
🚫
Non-Repudiation
Prevents a sender from later denying that they sent a message or performed an action. Provides proof of origin.
  • Mechanism: Digital signatures (RSA/DSA), Audit logs, Timestamps
  • Threat: Repudiation — denying a transaction or communication
  • Example: Digitally signed emails in court — sender cannot deny them
  • Legal: Indian IT Act 2000 recognises digital signatures legally
  • Tools: PKI, Notary services, Blockchain timestamps
🔐
Authorization
Determines what an authenticated user is permitted to do. Controls access to resources based on permissions.
  • Mechanism: ACL (Access Control Lists), RBAC (Role-Based), ABAC
  • Threat: Privilege escalation, Unauthorised access
  • Example: A bank employee can view accounts but not modify system config
  • Models: DAC (Discretionary), MAC (Mandatory), RBAC, ABAC
  • Principle: Least privilege — grant minimum necessary access
👤
Anonymity
The ability to use a system without revealing one's identity. Protects user privacy from observers and system operators.
  • Mechanism: Tor (onion routing), VPN, Zero-knowledge proofs
  • Use cases: Whistleblowing, Private browsing, Anonymous voting
  • Example: Tor browser routes traffic through 3 nodes, hiding origin IP
  • Trade-off: Anonymity vs Accountability — hard to balance
  • Privacy laws: GDPR (EU), PDPB (India) protect digital identity
TOPIC 2 · TYPES OF ATTACKS — COMPLETE COVERAGE
💥
DoS (Denial of Service) Attack
Attacker overwhelms a server/network with excessive traffic or requests, making it unavailable to legitimate users.
  • Goal: Availability disruption — not data theft
  • SYN Flood: Sends thousands of SYN packets, never completes TCP handshake. Server runs out of half-open connections
  • Ping of Death: Sends oversized ICMP packets causing buffer overflow and crash
  • UDP Flood: Bombards random ports with UDP packets; server responds to all with ICMP "unreachable"
  • Smurf Attack: Broadcasts ICMP ping with spoofed source IP (victim). All hosts reply to victim, overwhelming it
  • Prevention: Rate limiting, ingress filtering, SYN cookies, firewalls
🤖
DDoS (Distributed Denial of Service)
Multiple compromised machines (botnet) simultaneously attack a single target. Much harder to block than DoS.
  • Botnet: Network of infected devices (zombies) controlled by attacker (botmaster)
  • Scale: Can generate Tbps (terabits/sec) of attack traffic
  • Amplification: DNS/NTP amplification — small request generates huge response
  • Volumetric: Fills bandwidth with junk traffic
  • Protocol: Exploits weaknesses in network protocols
  • Application Layer: HTTP floods targeting web servers specifically
  • Mitigation: CDN, scrubbing centres, anycast diffusion, rate limiting
🎭
IP Spoofing
Attacker forges the source IP address in packets to impersonate another host or hide their identity.
  • How it works: Modify IP header's source address field before sending
  • Use in DoS: Spoof victim's IP so replies flood the victim (amplification)
  • Blind spoofing: Attacker can send but can't receive replies (different subnet)
  • Non-blind: Attacker on same subnet can see packets; predict sequence numbers
  • Prevention: Ingress filtering (ISPs drop packets with false source IPs), BCP38
  • Detection: Comparing TTL values, route consistency checks
🔁
Replay Attack
Attacker captures a valid network transmission and maliciously re-transmits it later to trick the receiving system.
  • Scenario: Intercept login packet → replay it later to authenticate as victim
  • Banking risk: Replay a "transfer $100" packet multiple times
  • Prevention 1 — Timestamps: Server rejects requests older than N seconds
  • Prevention 2 — Nonce: Include random one-time-use number in each request
  • Prevention 3 — Sequence numbers: Each packet numbered; duplicates rejected
  • Prevention 4 — Session tokens: One-time tokens invalidated after use
🕵️
MITM (Man-in-the-Middle) Attack
Attacker secretly positions between two communicating parties, intercepting and potentially altering messages without either party knowing.
  • Step 1: ARP poisoning — attacker sends fake ARP replies, linking their MAC to victim's IP
  • Step 2: All traffic now flows through attacker machine
  • Step 3: Attacker reads, modifies, or injects packets
  • SSL Stripping: Downgrades HTTPS to HTTP so content is readable
  • DNS Spoofing: Returns fake IP for domain lookups, redirecting users
  • Prevention: HTTPS/TLS, certificate pinning, HSTS, mutual authentication, VPN
👁️
Passive vs Active Attacks
Passive: Attacker observes data without modifying it. Active: Attacker modifies, injects, or disrupts data.
  • Passive — Eavesdropping: Sniffs network traffic with tools like Wireshark
  • Passive — Traffic Analysis: Even without content, patterns reveal information
  • Passive — Hard to detect — no modification means no obvious evidence
  • Active — Masquerade: Pretend to be a legitimate entity
  • Active — Modification: Alter messages in transit
  • Active — Fabrication: Insert counterfeit messages
  • Active — Interruption: DoS / disrupt availability
TOPIC 3 · GENERAL THREATS — WORMS, VIRUSES, TROJANS (FULL DETAIL)
🦠
Computer Viruses
A self-replicating program that attaches itself to legitimate programs or files. Requires human action to spread.
  • How it spreads: User executes infected file; virus copies itself to other files
  • Boot sector virus: Infects MBR; runs before OS loads
  • File infector: Attaches to .exe, .com, .dll files
  • Macro virus: Embedded in Word/Excel macros (e.g., Melissa virus)
  • Polymorphic virus: Changes its signature each replication to evade detection
  • Stealth virus: Hides modifications; reports original file size to OS
  • Payload: Corrupts/deletes files, logs keystrokes, encrypts data
  • Prevention: Antivirus, email filters, execution prevention (DEP)
🪱
Computer Worms
Self-contained malware that replicates and spreads automatically across networks without needing a host program or user interaction.
  • Key difference from virus: No host needed; spreads independently
  • Propagation: Exploits OS/service vulnerabilities (e.g., buffer overflow), email, shared drives
  • Morris Worm (1988): First major internet worm; exploited Unix sendmail, fingerd, rsh
  • ILOVEYOU (2000): Email worm; spread via attachment; affected 50M computers
  • WannaCry (2017): Exploited EternalBlue (Windows SMB); spread ransomware globally
  • Damage: Consumes bandwidth, crashes servers, delivers payloads
  • Prevention: Patch management, network segmentation, firewalls
🐴
Trojan Horses
Malware disguised as legitimate, useful software. Does not self-replicate. Tricks users into installing it voluntarily.
  • Named after: Greek myth — soldiers hid in a wooden horse gifted to Troy
  • RAT (Remote Access Trojan): Gives attacker full remote control of victim's PC
  • Backdoor Trojan: Creates persistent access point bypassing normal authentication
  • Banking Trojan: Steals online banking credentials (Zeus, SpyEye)
  • Dropper: Downloads and installs additional malware silently
  • Rootkit Trojan: Hides deep in OS, masks its own presence
  • Zeus Trojan: Stole credentials from millions; caused $70M+ in losses
  • Prevention: Software from trusted sources only, antivirus, sandboxing
💣
Ransomware
Malware that encrypts victim's files and demands payment (ransom) for the decryption key.
  • Mechanism: Asymmetric encryption — attacker holds private key
  • WannaCry (2017): Encrypted 300,000+ computers in 150 countries; NHS UK severely hit
  • NotPetya (2017): Disguised as ransomware; actually destructive wiper
  • Ransomware-as-a-Service: Criminal groups sell ransomware kits to affiliates
  • Prevention: Regular backups (offline), patch quickly, email filtering, EDR
  • Response: Do NOT pay (no guarantee); restore from backup
🔍
Spyware & Adware
Software that secretly collects user information and sends it to third parties without consent.
  • Keylogger: Records every keystroke (captures passwords, card numbers)
  • Screen capture: Takes screenshots periodically and transmits them
  • Browser hijacker: Redirects searches, changes homepage
  • Adware: Displays unwanted advertisements; may track browsing habits
  • Stalkerware: Hidden monitoring of partner's phone/PC (ethical/legal issues)
  • Prevention: Anti-spyware tools, browser security settings, MFA
💀
Rootkits & Botnets
Rootkit: Hides malware deep in the OS. Botnet: Network of infected machines controlled remotely.
  • Rootkit: Modifies OS kernel to hide processes, files, network connections
  • Kernel rootkit: Operates at ring-0 (highest privilege); very hard to remove
  • Bootkit: Infects bootloader; persists across OS reinstalls
  • Botnet — C&C: Command-and-Control server sends instructions to all bots
  • Mirai Botnet: Infected 300,000+ IoT devices; used for largest ever DDoS (620 Gbps)
  • Detection: Memory forensics, network traffic anomalies, behavioural analysis

TOPIC 1 · SYMMETRIC (SECRET KEY) CRYPTOGRAPHY — COMPLETE
🔑
DES — Data Encryption Standard
Published by NIST in 1977. First standardised encryption algorithm. Now considered insecure due to small key size.
  • Block size: 64 bits (processes 64-bit chunks)
  • Key size: 64 bits — but only 56 effective bits (8 bits are parity)
  • Structure: 16-round Feistel cipher network
  • Feistel structure: Each round: split block into L and R halves; R feeds through function F(R, subkey) XOR with L; then swap halves
  • Sub-keys: 16 × 48-bit sub-keys derived from 56-bit main key via key schedule
  • S-boxes: 8 substitution boxes provide non-linearity (confusion)
  • P-boxes: Permutation boxes provide diffusion (spread influence)
  • Why broken: 56-bit key = 2^56 = 72 quadrillion combinations; modern hardware cracks in <24h
  • EFF Deep Crack (1998): $250,000 machine; cracked DES in 56 hours, proving it unsafe
  • Modes: ECB (insecure), CBC (most used), CFB, OFB, CTR
🔑🔑🔑
Triple DES (3DES / TDEA)
Applies DES algorithm three times to each data block. Created as a stopgap replacement for DES when it became weak.
  • Operation: Encrypt(K1) → Decrypt(K2) → Encrypt(K3) = EDE mode
  • Why EDE? Allows backward compatibility — if K1=K2=K3, behaves like single DES
  • Key Option 1: K1≠K2≠K3 → 168-bit effective security (strongest)
  • Key Option 2: K1=K3≠K2 → 112-bit effective security
  • Key Option 3: K1=K2=K3 → same as DES (backward compat only)
  • Block size: Still 64-bit (same as DES)
  • Speed: 3× slower than DES; much slower than AES
  • Sweet32 attack: Birthday attack on 64-bit blocks; deprecated by NIST 2017
  • Legacy use: Old banking systems, ATM networks
⚡
AES — Advanced Encryption Standard
Selected by NIST in 2001 via competition. Based on Rijndael algorithm by Belgian cryptographers Joan Daemen and Vincent Rijmen. Current gold standard.
  • Block size: 128 bits (fixed)
  • Key sizes: 128-bit (10 rounds), 192-bit (12 rounds), 256-bit (14 rounds)
  • Structure: Substitution-Permutation Network (NOT Feistel)
  • State: 4×4 matrix of bytes (128 bits) called "state"
  • Round 1 — SubBytes: Each byte replaced using S-box (provides confusion)
  • Round 2 — ShiftRows: Rows of state matrix cyclically shifted left
  • Round 3 — MixColumns: Columns mixed using GF(2^8) arithmetic (provides diffusion)
  • Round 4 — AddRoundKey: XOR state with round key derived from key schedule
  • Final round: SubBytes + ShiftRows + AddRoundKey (no MixColumns)
  • Security: No practical attack known against full AES; used by NSA for top-secret data
  • Modes: CBC, CTR, GCM (authenticated encryption), CCM
🔐
Key Distribution Problem (Symmetric)
Major challenge: How do two parties securely share a secret key before they can communicate? Sending key over insecure channel defeats the purpose.
  • Problem: N users communicating = N(N-1)/2 keys needed → scales poorly
  • 100 users → 4,950 keys; 1000 users → 499,500 keys!
  • Solution 1 — KDC: Key Distribution Centre: trusted third party holds master keys; distributes session keys on request (Kerberos uses this)
  • Solution 2 — Diffie-Hellman: Key exchange over public channel without prior shared secret
  • Solution 3 — PKI: Use public key crypto to exchange symmetric keys (TLS/SSL does this)
  • Solution 4 — Out-of-band: Exchange key via different channel (phone, in-person)
  • Session keys: Use fresh key per session; even if one compromised, others safe
TOPIC 2 · ASYMMETRIC (PUBLIC KEY) CRYPTOGRAPHY — COMPLETE
🗝️
RSA — Rivest-Shamir-Adleman
Invented 1977 by Ron Rivest, Adi Shamir, Leonard Adleman at MIT. First practical public-key system. Security based on difficulty of integer factorisation.
  • Key generation: Choose 2 large primes p, q → n = p×q → φ(n)=(p-1)(q-1) → choose e coprime to φ(n) → d = e⁻¹ mod φ(n)
  • Public key: (e, n) — published to everyone
  • Private key: (d, n) — kept secret
  • Encryption: C = M^e mod n
  • Decryption: M = C^d mod n
  • Security: Factoring n to get p,q is computationally infeasible for large n
  • Key sizes: 1024-bit (weak), 2048-bit (current standard), 4096-bit (long-term)
  • Uses: HTTPS, SSH, S/MIME, PGP, code signing, digital signatures
  • Speed: 1000× slower than AES → used only for key exchange, not bulk data
📈
ECC — Elliptic Curve Cryptography
Public key crypto based on algebraic structure of elliptic curves over finite fields. Much smaller keys for equivalent security compared to RSA.
  • Math basis: Elliptic Discrete Logarithm Problem (ECDLP) — finding k in Q=kP
  • Key comparison: 256-bit ECC ≈ 3072-bit RSA in security strength
  • 160-bit ECC ≈ 1024-bit RSA; 384-bit ECC ≈ 7680-bit RSA
  • Benefits: Smaller keys → faster computation, less bandwidth, ideal for mobile/IoT
  • ECDH: Elliptic Curve Diffie-Hellman — key exchange using EC math
  • ECDSA: EC Digital Signature Algorithm — used in Bitcoin, TLS 1.3
  • Curves: P-256 (NIST), Curve25519 (Signal, WireGuard), secp256k1 (Bitcoin)
  • Uses: TLS 1.3, Bitcoin, Signal, mobile phones, smart cards, IoT
🤝
Diffie-Hellman Key Exchange
Invented 1976 by Whitfield Diffie and Martin Hellman. Allows two parties to establish a shared secret over a public, insecure channel without prior shared secret.
  • Security basis: Discrete Logarithm Problem — given g^x mod p, finding x is hard
  • Step 1: Alice and Bob publicly agree on prime p and generator g
  • Step 2: Alice picks secret a, computes A = g^a mod p, sends A to Bob
  • Step 3: Bob picks secret b, computes B = g^b mod p, sends B to Alice
  • Step 4: Alice computes S = B^a mod p = g^(ab) mod p
  • Step 5: Bob computes S = A^b mod p = g^(ab) mod p → Same S!
  • Eve sees: p, g, g^a mod p, g^b mod p — cannot compute g^(ab) mod p
  • Vulnerability: Susceptible to MITM if not authenticated (needs certificates)
  • DHE/ECDHE: Ephemeral versions provide Perfect Forward Secrecy (PFS)

TOPIC 3 · JAVA CRYPTOGRAPHY EXTENSIONS — FULL COVERAGE
☕
JCA — Java Cryptography Architecture
Core framework defining architecture and API for cryptographic services in Java. Part of JDK since Java 1.1.
  • Provider architecture: Pluggable security providers (Sun, BouncyCastle, etc.)
  • Engine classes: MessageDigest, Signature, KeyPairGenerator, KeyFactory
  • Algorithm independence: Code doesn't need to change when algorithm changes
  • Provider independence: Can swap security providers without code changes
  • MessageDigest: Provides SHA-256, MD5, SHA-512 hash algorithms
  • Signature: Provides RSA, DSA, ECDSA digital signatures
🔒
JCE — Java Cryptography Extension
Extends JCA with encryption, key agreement, and MAC support. Provides symmetric cipher functionality.
  • Cipher class: Implements AES, DES, 3DES, RSA encryption/decryption
  • KeyGenerator: Generates symmetric keys (AES, DES)
  • Mac: Implements HMAC-SHA256, HMAC-SHA512
  • SecretKeyFactory: Converts between key representations
  • KeyAgreement: Implements Diffie-Hellman key exchange
  • Export restrictions: Originally restricted; unlimited jurisdiction policy files needed
🔐
JSSE — Java Secure Socket Extension
Provides Java implementation of SSL and TLS protocols for secure network communication.
  • SSLSocket: SSL/TLS-enabled socket for secure communication
  • SSLServerSocket: Server-side SSL socket
  • SSLContext: Represents SSL/TLS protocol; creates socket factories
  • KeyManager: Manages key material for authentication
  • TrustManager: Manages certificate trust decisions
  • Supports: TLS 1.2, TLS 1.3, mutual authentication, SNI
  • Use case: All Java HTTPS connections (HttpsURLConnection) use JSSE
🎫
JGSS — Java Generic Security Services
Java binding for GSS-API (RFC 2853). Provides security services in a mechanism-independent way. Main use: Kerberos integration.
  • GSS-API: Generic Security Services Application Program Interface
  • Mechanism independent: Same API works for Kerberos, SPNEGO, etc.
  • GSSContext: Represents security context between communicating parties
  • GSSCredential: Holds security credentials (Kerberos tickets)
  • GSSName: Represents principal in a security context
  • Use case: Enterprise Java apps authenticating to Kerberos (Active Directory)
  • JAAS integration: Works with JAAS for login authentication
TOPIC 4 · CRYPTOGRAPHIC ATTACKS
💪
Brute Force Attack
Try every possible key until the correct one is found. Always works eventually — question is whether it's feasible in time.
  • DES (56-bit): 2^56 = 7.2×10^16 combinations → feasible with hardware
  • AES-128: 2^128 = 3.4×10^38 → all computers in universe can't break before sun dies
  • GPU/ASIC: Graphics cards and custom chips dramatically speed up brute force
  • Defence: Use longer keys (≥128-bit), rate limiting, account lockout
🎂
Birthday Attack
Exploits the Birthday Paradox: in a group of 23 people, 50% chance two share a birthday. Applied to hash collisions.
  • Principle: For n-bit hash, find collision in 2^(n/2) operations
  • MD5 (128-bit): 2^64 operations to find collision (found in practice!)
  • SHA-1 (160-bit): 2^80 theoretical → Google's SHAttered attack 2017 proved collision
  • Defence: Use SHA-256 or SHA-3; larger hash output size
  • Impact: Attacker can create two docs with same hash → forge signatures
🔬
Differential & Linear Cryptanalysis
Statistical attacks on block ciphers that analyse how differences in plaintext pairs affect ciphertext.
  • Differential: Input pair differences → output pair differences → reveals key bits
  • Linear: Find linear approximations between plaintext, ciphertext, key bits
  • Known-plaintext attack: Attacker has plaintext-ciphertext pairs
  • Chosen-plaintext: Attacker can encrypt any plaintext of their choice
  • DES and these attacks: DES was designed with differential cryptanalysis in mind by IBM/NSA
  • AES resistance: Wide-trail strategy provides provable resistance

TOPIC 1 · HASH FUNCTIONS — MD5, SHA FAMILY — COMPLETE
#️⃣
Hash Functions — Core Concepts
A hash function takes variable-length input (message) and produces a fixed-length output (hash/digest/fingerprint). One-way function — cannot reverse.
  • Formal: H(M) = h where M is any length, h is fixed length
  • Deterministic: Same input ALWAYS gives same output
  • Fixed output size: 1 byte or 1 GB input → same length hash
  • Fast to compute: O(n) — linear in input size
  • Avalanche effect: Change 1 bit of input → ~50% of output bits change
  • Applications: File integrity, passwords, digital signatures, HMAC, blockchain
  • NOT encryption: Hashing is one-way; encryption is two-way
  • Password storage: Never store passwords — store H(password+salt)
🔏
4 Properties of Hash Functions (MUST KNOW)
All cryptographic hash functions must satisfy these four security properties to be considered secure.
  • 1. Pre-image Resistance (One-way): Given h, computationally infeasible to find M such that H(M)=h. Even knowing the hash, you can't find the original input.
  • 2. Second Pre-image Resistance (Weak collision): Given M1, computationally infeasible to find different M2 such that H(M1)=H(M2). Can't find another message with same hash as a given message.
  • 3. Collision Resistance (Strong collision): Computationally infeasible to find ANY two different messages M1≠M2 such that H(M1)=H(M2). Harder than weak collision — can choose any pair.
  • 4. Avalanche Effect: A tiny change in input (even 1 bit) produces a drastically different hash (~50% bits change). Prevents small modifications from going undetected.
5️⃣
MD5 — Message Digest Algorithm 5
Designed by Ron Rivest in 1992. Produces 128-bit (32 hex character) hash. Now cryptographically broken — should never be used for security purposes.
  • Output: 128-bit (16 bytes = 32 hex characters)
  • Structure: Merkle-Damgård construction; processes 512-bit blocks
  • 4 rounds of 16 operations each
  • Wang's attack (2004): Chinese researcher Xiaoyun Wang found MD5 collisions in minutes on a laptop
  • Flame malware (2012): Used MD5 collision to forge Microsoft Windows Update certificate
  • Still used for: Non-security checksums, file deduplication, database indexing
  • Never use for: Password storage, certificates, digital signatures
🔐
SHA Family — SHA-1, SHA-2, SHA-3
Secure Hash Algorithm family developed by NSA/NIST. SHA-2 (SHA-256/SHA-512) is current standard. SHA-3 is the alternative standard.
  • SHA-1: 160-bit output; 80 rounds; deprecated. Google's SHAttered attack (2017) produced first SHA-1 collision using 9.2×10^18 computations
  • SHA-224: Truncated SHA-256 → 224-bit output
  • SHA-256: 256-bit output; 64 rounds; used in Bitcoin, TLS 1.2/1.3, HTTPS certificates
  • SHA-384: Truncated SHA-512 → 384-bit output
  • SHA-512: 512-bit output; 80 rounds; used for highest security applications
  • SHA-3 (Keccak): Completely different design — sponge construction; not vulnerable to SHA-2's weaknesses; backup standard
  • NIST competition (2007-2012): 64 candidates entered; Keccak by Belgian team won

TOPIC 2 · MAC — MESSAGE AUTHENTICATION CODE — COMPLETE
🔏
MAC — Message Authentication Code
A MAC is a short piece of information used to authenticate a message — confirming it came from the stated sender (authentication) and hasn't been altered (integrity). Requires a shared secret key.
  • Formula: MAC = f(Key, Message) where f is a keyed hash function
  • Sender: Computes MAC using shared key K → sends (Message, MAC)
  • Receiver: Recomputes MAC using K and received message → compare
  • If MAC matches: Message is authentic and unaltered
  • Security: Attacker without K cannot forge a valid MAC
  • Provides: Message integrity + Authentication
  • Does NOT provide: Non-repudiation (both parties share same key)
  • Does NOT provide: Confidentiality (message still readable)
🔐
HMAC — Hash-Based MAC
The most widely used MAC construction. Uses a cryptographic hash function combined with a secret key. Formally defined in RFC 2104.
  • Formula: HMAC(K,M) = H( (K⊕opad) || H( (K⊕ipad) || M ) )
  • ipad: Inner padding = 0x36 repeated; opad = Outer padding = 0x5C repeated
  • Two-pass: Inner hash: H(K⊕ipad || M); Outer hash: H(K⊕opad || inner)
  • Security: Even if underlying hash has weakness, HMAC provides stronger guarantee
  • HMAC-MD5: 128-bit output (avoid for new systems)
  • HMAC-SHA256: 256-bit output (current standard)
  • HMAC-SHA512: 512-bit output (highest security)
  • Uses: TLS, IPSec, SSH, JWT (JSON Web Tokens), API authentication
🏦
CBC-MAC (Cipher Block Chaining MAC)
MAC constructed using a block cipher in CBC mode. Message encrypted in CBC mode; final ciphertext block is the MAC.
  • Process: Break M into blocks → CBC encrypt with key K → last block = MAC
  • Used in: Banking (ISO 9797), smart cards, EMV chip cards
  • CMAC: Improved version of CBC-MAC; fixes padding vulnerabilities
  • Weakness: Raw CBC-MAC vulnerable to extension attacks → use CMAC
  • GMAC: Galois MAC — component of AES-GCM authenticated encryption
TOPIC 3 · DIGITAL SIGNATURES — RSA AND DSA — COMPLETE
✍️
Digital Signature — Concepts & Purpose
A mathematical scheme that provides authentication, integrity, and non-repudiation for digital documents. The digital equivalent of a handwritten signature but much more secure.
  • Provides 3 services: Authentication (who signed), Integrity (not modified), Non-Repudiation (can't deny)
  • Signing: Sender hashes message → encrypts hash with PRIVATE key → this is the signature
  • Verification: Receiver decrypts signature with sender's PUBLIC key → compare with recomputed hash
  • Why hash first? Signing is slow (asymmetric crypto); hash is small fixed-size; sign the hash instead
  • Legal status: IT Act 2000 (India), eIDAS (EU), ESIGN Act (US) — legally equivalent to handwritten signature
  • NOT encryption: Digital signature ≠ encrypted message. Signature just proves authenticity.
🔑
RSA Digital Signature — Mechanism
Uses RSA asymmetric cryptography. Sender signs with private key; anyone can verify with corresponding public key.
  • Sign (Alice): h = SHA256(M) → S = h^d mod n (encrypt hash with private key d)
  • Send: Alice sends (M, S) to Bob
  • Verify (Bob): h' = SHA256(M) → h = S^e mod n (decrypt with Alice's public e) → if h'==h → valid
  • If tampered: S^e mod n ≠ SHA256(M') → invalid → tampering detected
  • Key usage (opposite of encryption!): Sign with PRIVATE, Verify with PUBLIC
  • RSA-PSS: Probabilistic Signature Scheme — current secure RSA signature standard
  • RSA-PKCS1 v1.5: Older scheme; has padding oracle vulnerabilities
🖊️
DSA — Digital Signature Algorithm
US Federal standard for digital signatures (FIPS 186). Based on discrete logarithm problem. Designed specifically for signing only — cannot be used for encryption.
  • Parameters: p (prime), q (160-bit prime dividing p-1), g (generator), x (private), y=g^x mod p (public)
  • Signing: k = random nonce; r = (g^k mod p) mod q; s = k⁻¹(H(M)+xr) mod q; signature = (r,s)
  • Verification: w=s⁻¹; u1=H(M)w mod q; u2=rw mod q; v=(g^u1 × y^u2 mod p) mod q; valid if v=r
  • Critical: k must be unique random per signature. Sony PS3 used fixed k → private key extracted!
  • DSA vs RSA: DSA faster to sign; RSA faster to verify; RSA can encrypt too
  • ECDSA: DSA on elliptic curves — smaller keys, used in Bitcoin, TLS 1.3

TOPIC 4 · BIOMETRIC AUTHENTICATION — COMPLETE
👆
Biometric Authentication — Concepts
Authentication based on unique biological characteristics of a person. The third factor: "something you are" (vs password=know, token=have).
  • Physiological biometrics: Fingerprint, iris, retina, face, hand geometry, DNA, vein patterns
  • Behavioural biometrics: Keystroke dynamics, signature, voice, gait (walking pattern)
  • Enrollment: User's biometric scanned and stored as template (not raw image)
  • Matching: New scan compared against stored template using similarity threshold
  • Threshold: Must set balance between FAR and FRR
📊
FAR, FRR, EER — Biometric Metrics
Key performance metrics for biometric systems that define accuracy and security trade-offs.
  • FAR (False Accept Rate): Probability impostor is incorrectly accepted. Security metric — lower = more secure
  • FRR (False Reject Rate): Probability legitimate user is incorrectly rejected. Usability metric — lower = more convenient
  • EER (Equal Error Rate): Point where FAR = FRR. Lower EER = better biometric system
  • Trade-off: Increase threshold → lower FAR (more secure) but higher FRR (less convenient)
  • High-security scenario: Airport immigration → set low FAR (accept few impostors), tolerate higher FRR
  • Convenience scenario: Phone unlock → lower threshold, accept higher FAR
👁️
Types of Biometrics
Different biometric modalities with their characteristics, accuracy, and real-world applications.
  • Fingerprint: Most common; unique even in identical twins; used in phones, police, Aadhaar. FAR ~0.001%
  • Iris recognition: 266 degrees of freedom (vs 13 for fingerprint); very accurate; unchanged for life; used in border control, UAE national ID
  • Facial recognition: Non-contact; can be fooled by photos (2D); 3D IR facial scan (iPhone Face ID) is very secure; used in police surveillance, phone unlock
  • Retina scan: Blood vessel pattern in eye; highly accurate; intrusive (close contact); used in high-security facilities
  • Voice recognition: Can change with illness/age; affected by noise; used in call centres, banking IVR
  • Vein pattern: Near-infrared camera captures vein patterns; liveness detection built-in; used in Japanese ATMs
⚠️
Biometric Vulnerabilities & Issues
Biometrics have unique security and privacy concerns not present in password systems.
  • Cannot change: If biometric compromised, cannot "reset" like a password. This is permanent.
  • Spoofing: Fingerprint can be lifted from glass and reproduced in gelatin; photo can fool 2D face systems
  • Database breach: Stolen biometric templates are permanent identity theft
  • Liveness detection: Anti-spoofing measure — checks if biometric is from live person (pulse, 3D depth, blinking)
  • Privacy: Biometric data is most sensitive personal data; GDPR classifies as "special category"
  • Template protection: Store fuzzy hashes or encrypted templates; never raw biometric data
  • Aadhaar breach (2018): Database of 1.1 billion Indian biometrics reportedly leaked — permanent identity risk

TOPIC 1 · PKI — PUBLIC KEY INFRASTRUCTURE — COMPLETE
🏛️
PKI Overview & Architecture
A framework of policies, procedures, hardware, software, and people to manage the lifecycle of digital certificates and public key encryption. The trust backbone of the internet.
  • Problem PKI solves: How do you trust a public key you receive? Anyone could send any public key claiming to be a bank. PKI ties public keys to verified identities.
  • CA (Certificate Authority): Trusted third party that issues signed digital certificates. Examples: DigiCert, Comodo, Let's Encrypt, GlobalSign
  • RA (Registration Authority): Verifies identity of certificate applicant before CA issues cert. Offloads identity verification from CA
  • Repository: Directory (LDAP) where certificates and CRLs are published
  • Trust anchor: Root CA certificate — pre-installed in OS and browsers
  • End entity: The certificate subject (person, server, device)
  • PKI in numbers: ~1000 trusted root CAs globally; billions of certificates in use; ~90% of web traffic secured by PKI-based TLS
📜
Digital Certificates — X.509 Standard
An X.509 digital certificate is an electronic document that uses a digital signature to bind a public key with an identity. The de facto standard for PKI certificates.
  • Version: X.509 v3 is current standard
  • Serial number: Unique identifier assigned by CA
  • Algorithm identifier: Algorithm used to sign certificate (e.g., SHA256WithRSAEncryption)
  • Issuer: Distinguished Name (DN) of the CA that issued cert
  • Validity period: Not Before, Not After dates
  • Subject: DN of entity the cert belongs to (CN=www.google.com, O=Google LLC, C=US)
  • Subject Public Key Info: The public key + algorithm
  • Extensions (v3): SAN (Subject Alternative Names), KeyUsage, BasicConstraints, etc.
  • CA's Digital Signature: CA signs all above fields with its private key
  • Certificate types: DV (Domain Validated), OV (Org Validated), EV (Extended Validation)
🔗
Certificate Trust Chain (Chain of Trust)
Hierarchical trust model where each certificate is signed by a higher-level CA, ultimately leading to a Root CA that is trusted by all parties.
  • Root CA: Self-signed certificate; ultimate trust anchor; private key kept offline in HSM; 20-30 year validity
  • Intermediate CA: Signed by Root CA; used for day-to-day signing; more revocable than Root
  • End-entity certificate: Your website/email cert; signed by Intermediate CA; 1-2 year validity
  • Chain validation: Browser checks: End-entity signed by Intermediate? Intermediate signed by Root? Root in trust store?
  • Why intermediate? Isolates Root CA from internet; if Intermediate compromised, revoke it without revoking Root
  • Trust store: OS/browser pre-installs ~150 trusted Root CA certificates
  • Example: amazon.com cert → signed by Amazon RSA 2048 M02 (Intermediate) → signed by DigiCert (Root)
🚫
CRL & OCSP — Certificate Revocation
Mechanism to invalidate compromised or expired certificates before their natural expiry date.
  • CRL (Certificate Revocation List): Published list of serial numbers of revoked certificates, signed by CA
  • CRL drawback: Can be large; clients download periodically → stale data possible
  • OCSP (Online Certificate Status Protocol): Real-time query to CA: "Is this cert revoked?" → "good/revoked/unknown"
  • OCSP Stapling: Server proactively fetches and caches its own OCSP response; includes it in TLS handshake; browser doesn't need to query CA
  • Revocation reasons: Key compromise, CA compromise, affiliation changed, superseded, cessation of operation
  • CRLite: Mozilla's compressed CRL for Firefox — full revocation data in ~10MB
🎯
POP Key Interface (Proof of Possession)
Mechanism that allows an entity to prove it actually has possession of the private key corresponding to the public key in a certificate request.
  • Problem: Anyone could submit Alice's public key in a certificate request. POP proves the requester holds the matching private key.
  • Mechanism 1 — Signature-based: Sign a challenge or the certificate request itself with the private key. CA verifies with submitted public key.
  • Mechanism 2 — Key transport: CA encrypts secret with submitted public key; requester decrypts with private key
  • Mechanism 3 — Key agreement: Diffie-Hellman exchange proves possession
  • PKCS#10 (CSR): Certificate Signing Request includes signature proving private key possession
  • Used in: CA enrollment, SCEP, EST, ACME protocols
TOPIC 2 · FIREWALLS — ALL TYPES — COMPLETE
🧱
Packet Filtering Firewall
Inspects each packet independently based on predefined rules (ACLs). Checks source/destination IP and port. Stateless — no memory of connections.
  • Operates at: Network layer (Layer 3) and Transport layer (Layer 4)
  • Inspects: Source IP, Destination IP, Source Port, Dest Port, Protocol (TCP/UDP/ICMP)
  • Stateless: Each packet treated independently — doesn't track connection state
  • Fast: Simple comparison against rules; very high throughput
  • Weakness: Cannot detect fragmented packet attacks; doesn't understand connection context; vulnerable to IP spoofing
  • Example rule: DENY TCP * * 23 (Block all Telnet traffic on port 23)
  • Used in: Routers (Cisco ACLs), basic network segmentation
🔍
Stateful Inspection Firewall
Tracks the state of network connections. Maintains a state table of active connections and makes decisions based on connection context, not just individual packets.
  • State table: Records source IP, dest IP, port, protocol, connection state (SYN_SENT, ESTABLISHED, etc.)
  • Benefit: Blocks packets that claim to be part of an established connection that doesn't exist (SYN spoofing)
  • Benefit: Allows return traffic automatically — no rule needed for responses
  • TCP tracking: Validates TCP sequence numbers; detects session hijacking attempts
  • Still stateful Layer 4: Doesn't inspect application payload
  • Examples: Windows Firewall, iptables (Linux), Cisco PIX/ASA
  • Most common firewall type for enterprise perimeter
🎭
Application Layer Firewall (Proxy)
Also called proxy firewall or application gateway. Operates at Layer 7 (Application layer). Understands application protocols — can inspect HTTP, FTP, DNS content.
  • Acts as proxy: Client connects to firewall; firewall connects to server; no direct connection
  • Deep inspection: Can read HTTP headers, URLs, DNS queries, FTP commands
  • Can block: Malicious URLs, SQL injection attempts, XSS in web traffic
  • Can enforce: Specific web policies (block social media, allow only GET requests)
  • WAF (Web Application Firewall): Specialised for HTTP/HTTPS; protects web apps from OWASP Top 10
  • Drawback: Slower than packet filtering; requires protocol-specific proxies
  • Examples: Squid proxy, Cloudflare WAF, AWS WAF, F5 BIG-IP
🚀
NGFW — Next-Generation Firewall
Combines traditional firewall with additional security features: DPI (Deep Packet Inspection), IPS, application awareness, user identity, SSL inspection, and threat intelligence.
  • DPI: Reads packet payloads to identify applications (not just by port)
  • IPS integration: Intrusion Prevention System — blocks known attack signatures
  • Application awareness: Identifies Facebook, WhatsApp, BitTorrent regardless of port
  • User-based policy: "Block social media for Sales dept but allow for Marketing"
  • SSL/TLS inspection: Decrypts HTTPS traffic, inspects, re-encrypts
  • Threat intelligence: Real-time updates of malicious IPs, domains, file hashes
  • Examples: Palo Alto Networks, Fortinet FortiGate, Cisco Firepower, Check Point

TOPIC 3 · SMART CARDS, ZERO-KNOWLEDGE PROOFS & ATTACKS
💳
Smart Cards — Architecture & Types
A tamper-resistant hardware token (credit-card sized) with embedded microprocessor and memory. Can perform cryptographic operations internally, never exposing secret keys.
  • Components: CPU, RAM (~1-8KB), ROM (~32-64KB), EEPROM/Flash (~32-256KB), crypto coprocessor
  • Contact smart card: ISO 7816; physical gold contacts for power and data; inserted into reader
  • Contactless (RFID/NFC): ISO 14443; communicates via electromagnetic field; tap to use
  • Dual interface: Both contact and contactless (modern bank cards)
  • Java Card: Subset of Java for smart cards; runs Java Card applets
  • EMV: Europay-Mastercard-Visa standard for chip-and-PIN payment cards
  • APDU: Application Protocol Data Units — command-response protocol for smart card communication
  • SIM cards: A type of smart card; stores subscriber identity, authenticate to GSM/LTE network
🧙
Zero Knowledge Proof (ZKP)
A cryptographic protocol where one party (prover) can convince another party (verifier) that they know a secret, without revealing any information about the secret itself.
  • 3 properties of ZKP: Completeness (honest prover always convinces), Soundness (dishonest prover rarely convinces), Zero-knowledge (verifier learns nothing except "true")
  • Cave analogy: Ring-shaped cave with door. Verifier waits at entrance. Prover enters. Verifier calls which side to exit. Prover exits correct side (using door if needed). Repeat 20 times → if prover always exits correct side, they know the door secret. Verifier never saw the door.
  • Interactive ZKP: Multiple rounds of challenge-response (as above)
  • Non-interactive ZKP (NIZK): Single message; uses hash functions; used in zk-SNARKs (Zcash cryptocurrency)
  • Smart card use: Card proves knowledge of PIN to terminal without transmitting PIN; terminal can't learn PIN from protocol
  • Modern uses: Privacy-preserving authentication, anonymous credentials, blockchain privacy (Zcash, zkRollups)
⚔️
Attacks on Smart Cards
Smart cards face unique physical and logical attack categories. Hardware security is multi-layered defence against sophisticated attackers.
  • Side-Channel Attack: Exploit physical emanations rather than mathematical weaknesses
  • — Power Analysis: DPA (Differential Power Analysis) — measure power consumption during crypto operations; correlate with secret key. Kocher et al. (1999) extracted DES keys from smart cards
  • — Timing Attack: Measure execution time; different operations take different time; reveals key
  • — EM Analysis: Electromagnetic emissions reveal internal operations
  • Fault Injection: Deliberately introduce errors to bypass security checks
  • — Voltage glitching: Spike power supply to cause instruction skip (skip PIN check!)
  • — Laser fault injection: Hit chip with laser at precise time to flip bits
  • Physical Attacks: Micro-probing (attach probe to internal bus), reverse engineering (dissolve packaging with acid, image chip)
  • Software Attacks: Protocol attacks, buffer overflow in card OS, malicious applets (Java Card sandboxing)
  • Relay Attack (contactless): Extend card's NFC range — attacker near you, accomplice at payment terminal — card thinks it's tapping terminal

TOPIC 4 · VPN — VIRTUAL PRIVATE NETWORK
🌐
VPN — Overview & Purpose
Creates an encrypted tunnel over a public network (internet) allowing secure communication as if parties were on a private network.
  • How it works: Wrap (encapsulate) original packet in encrypted outer packet; send over internet; decrypt at destination
  • Site-to-site VPN: Connect two office networks; router-to-router; always on
  • Remote access VPN: Employee connects to office network from home; client software required
  • Benefits: Encrypts traffic on public WiFi; hides true IP; accesses geo-restricted content; secures remote work
  • IPSec VPN: Network-layer encryption; site-to-site standard; high performance
  • SSL/TLS VPN: Browser-based; port 443; easier through firewalls; remote access
  • WireGuard: Modern, fast, simple VPN protocol using ChaCha20+ECDH; used in many consumer VPNs

TOPIC 1 · KERBEROS AUTHENTICATION — FULL MECHANISM
🐕
Kerberos — Concepts & Architecture
Network authentication protocol developed at MIT. Uses symmetric-key cryptography and a trusted third party (KDC) to provide authentication. Named after the 3-headed dog guarding Hades in Greek mythology.
  • Goal: Single sign-on — log in once, access all authorised services
  • KDC (Key Distribution Centre): Trusted third party; contains AS + TGS; central to all authentication
  • AS (Authentication Server): Verifies user identity; issues TGT
  • TGS (Ticket Granting Server): Issues service-specific tickets
  • TGT (Ticket Granting Ticket): Proof of identity; used to request service tickets; time-limited (typically 8-10 hours)
  • Service Ticket: Grants access to specific service; contains session key
  • Realm: Administrative domain (e.g., COMPANY.COM); multiple realms can trust each other
  • Encryption used: AES (modern Kerberos 5); DES (old Kerberos 4 — insecure)
  • Windows Active Directory: Kerberos 5 is default authentication protocol since Windows 2000
🎫
Kerberos — Step-by-Step Authentication (6 Steps)
Complete authentication flow from initial login to service access. Client never sends password over network.
  • Step 1 (Client → AS): Client sends username + timestamp (to prevent replay) in plaintext
  • Step 2 (AS → Client): AS sends: (a) TGT encrypted with TGS's secret key (client can't read) + (b) Session key K_TGS encrypted with client's password-derived key
  • Step 3 (Client decrypts): Client uses password to decrypt K_TGS; holds TGT opaquely
  • Step 4 (Client → TGS): Client sends TGT + Authenticator (timestamp encrypted with K_TGS) + requested service name
  • Step 5 (TGS → Client): TGS decrypts TGT with its key; verifies authenticator; sends: Service Ticket (encrypted with service's key) + Service Session Key K_S
  • Step 6 (Client → Service): Client presents Service Ticket + new Authenticator (encrypted with K_S); Service decrypts ticket; verifies authenticator → access granted!
  • Client's password never sent over network at any step

TOPIC 2 · SSL / TLS — COMPLETE PROTOCOL EXPLANATION
🔒
SSL/TLS — Overview & History
SSL (Secure Sockets Layer) and its successor TLS (Transport Layer Security) provide cryptographic security for communications over a computer network. The foundation of HTTPS.
  • SSL 2.0 (1995): Netscape; multiple vulnerabilities; deprecated
  • SSL 3.0 (1996): Fixed SSL 2.0 issues; vulnerable to POODLE (2014); deprecated
  • TLS 1.0 (1999): Based on SSL 3.0; vulnerable to BEAST, POODLE; deprecated 2021
  • TLS 1.1 (2006): Minor fixes; deprecated 2021 by major browsers
  • TLS 1.2 (2008): Added SHA-256, AES-GCM, AEAD; still widely used
  • TLS 1.3 (2018): Major redesign; removed weak algorithms; 1-RTT handshake; 0-RTT resumption; mandatory PFS; current standard
  • Services: Confidentiality (encryption), Integrity (MAC), Authentication (certificates)
🤝
TLS Handshake — Step by Step (TLS 1.2)
The negotiation phase that establishes session parameters before encrypted data exchange begins.
  • Step 1 — ClientHello: Client sends TLS version, random nonce, list of supported cipher suites, compression methods
  • Step 2 — ServerHello: Server selects cipher suite, sends its random nonce, session ID
  • Step 3 — Certificate: Server sends X.509 certificate (contains public key)
  • Step 4 — ServerHelloDone: Server signals end of hello messages
  • Step 5 — ClientKeyExchange: Client sends Pre-Master Secret encrypted with server's public key (RSA) OR Diffie-Hellman parameters
  • Step 6 — Key derivation: Both derive Master Secret from Pre-Master Secret + both random nonces → generate 4 keys (client/server write key + MAC key)
  • Step 7 — ChangeCipherSpec: Both signal switch to encrypted communication
  • Step 8 — Finished: Both send encrypted hash of all handshake messages — verifies integrity of handshake
⚡
TLS 1.3 — Improvements
TLS 1.3 is a major redesign that removes weak algorithms, speeds up handshake, and mandates Perfect Forward Secrecy.
  • 1-RTT handshake: TLS 1.2 needed 2 RTT; TLS 1.3 needs only 1 RTT (saves 50ms per connection)
  • 0-RTT resumption: For returning sessions, data sent in first message (trade-off: replay risk)
  • Mandatory PFS: Only ECDHE/DHE key exchange allowed — every session has unique key
  • Removed: RSA key exchange, static DH, weak ciphers (RC4, DES, 3DES), MD5/SHA-1, compression
  • Cipher suites reduced: Only 5 AEAD cipher suites (AES-GCM, AES-CCM, ChaCha20-Poly1305)
  • Encrypted handshake: Certificate and most handshake fields encrypted (TLS 1.2 sends certificate in plaintext)
  • SNI encryption: ESNI/ECH hides hostname from network observers
🔐
TLS Cipher Suites
A cipher suite specifies the exact algorithms used for key exchange, authentication, encryption, and message authentication in a TLS session.
  • Format: TLS_[KeyExchange]_[Authentication]_WITH_[Cipher]_[HashMAC]
  • Example: TLS_ECDHE_RSA_WITH_AES_256_GCM_SHA384 means: ECDHE key exchange, RSA auth, AES-256-GCM encryption, SHA-384 MAC
  • TLS 1.3 suites: TLS_AES_256_GCM_SHA384, TLS_CHACHA20_POLY1305_SHA256
  • Recommended: Any suite with ECDHE (PFS) + AES-GCM (AEAD) + SHA-256/384
  • Avoid: NULL (no encryption!), RC4, 3DES, CBC (padding oracle risk), export ciphers
  • AEAD: Authenticated Encryption with Associated Data — provides encryption + integrity in one operation
TOPIC 3 · IPSec — IP SECURITY COMPLETE
📦
IPSec — Architecture & Overview
A suite of protocols for securing IP communications by authenticating and/or encrypting each IP packet. Operates at Network layer (Layer 3) — transparent to applications above.
  • Components: AH (Authentication Header), ESP (Encapsulating Security Payload), IKE (Internet Key Exchange)
  • SA (Security Association): One-directional agreement on algorithms and keys; identified by SPI (Security Parameter Index)
  • SAD (Security Association Database): Stores all active SAs
  • SPD (Security Policy Database): Defines which traffic to protect and how
  • Modes: Transport (payload only) and Tunnel (entire packet)
  • IKE: Internet Key Exchange — negotiates SAs and key material; uses DH
  • IKEv2: Improved IKE with EAP authentication, MOBIKE (mobility), fewer round trips
🔏
AH — Authentication Header
IPSec protocol providing data integrity and origin authentication for IP packets. Does NOT provide confidentiality (no encryption). Added as an IP extension header.
  • Provides: Integrity + Authentication (both)
  • Does NOT provide: Confidentiality — payload readable
  • How it works: Computes HMAC over IP header fields + payload → includes in AH header
  • Covers: Immutable IP header fields + entire payload
  • Protocol number: 51
  • Problem with NAT: NAT modifies IP header (changes source IP) → breaks AH integrity check → AH incompatible with NAT traversal
  • Use case: When you need to prove a packet really came from a specific source and wasn't modified, but content is not sensitive
🔒
ESP — Encapsulating Security Payload
IPSec protocol providing confidentiality, integrity, and authentication. The most commonly used IPSec protocol. Encrypts and/or authenticates the payload.
  • Provides: Confidentiality (encryption) + Integrity + Authentication
  • Protocol number: 50
  • Algorithms: AES-128/256-CBC (encryption), HMAC-SHA256/SHA512 (integrity)
  • Modern: AES-GCM provides both encryption and integrity in one operation (AEAD)
  • Transport mode: Encrypts only payload; IP header remains plaintext
  • Tunnel mode: Encrypts entire original IP packet; adds new outer IP header
  • NAT-T: ESP can be encapsulated in UDP port 4500 for NAT traversal
  • Anti-replay: Sequence number field prevents replay attacks
🔄
Transport Mode vs Tunnel Mode
Two modes determine how much of the IP packet is protected by IPSec.
  • Transport Mode: Only the payload is protected; original IP header preserved; source and destination IPs visible
  • Transport use: Host-to-host communication (e.g., remote admin, server-to-server)
  • Tunnel Mode: Entire original IP packet (header + payload) is encapsulated and protected; new outer IP header added
  • Tunnel use: VPN gateways; site-to-site VPN; hides internal IP addressing
  • Example Tunnel: Mumbai office (10.0.1.0/24) ↔ Delhi office (10.0.2.0/24); outer packet shows VPN gateway IPs; inner packet shows internal IPs
  • ESP Tunnel is most common IPSec VPN configuration

TOPIC 4 · E-COMMERCE SECURITY — SET, e-CASH, MICROPAYMENTS
💳
SET — Secure Electronic Transaction
Protocol developed jointly by Visa and Mastercard in 1997 for securing credit card transactions over the internet. Uses PKI and dual signatures.
  • Goal: Ensure merchant never sees full credit card number; bank never sees purchase details
  • Parties: Cardholder (customer), Merchant, Payment Gateway, Issuer (cardholder's bank), Acquirer (merchant's bank), Certificate Authority
  • Dual Signature: Core innovation of SET
  • — Customer creates: OI (Order Info for merchant) + PI (Payment Info for bank)
  • — Hashes: H(OI) and H(PI) → POMD = H(H(OI) || H(PI))
  • — Signs POMD with customer's private key
  • — Merchant sees OI + H(PI) — can verify order but NOT see card number
  • — Bank sees PI + H(OI) — can verify payment but NOT see order details
  • All parties receive certificates from CA
  • Why not widely adopted: Complexity, required cardholder software installation; replaced by 3D Secure (Verified by Visa)
💰
Electronic Cash (e-Cash)
Digital currency that replicates the anonymity properties of physical cash. Users spend without revealing identity. Invented by David Chaum (DigiCash, 1989).
  • Core property: Anonymity — bank cannot trace spending to specific buyer
  • Blind signature: Bank signs coin without seeing its serial number; user unblinds and spends it
  • Blind signature process: User picks random serial r; computes m'=m·r^e mod n; bank signs m'; user divides by r to get signed coin; bank signed it without knowing m!
  • Double-spending: Digital cash can be copied; prevention requires online verification with bank or cryptographic protocols
  • DigiCash (1989-1998): First e-cash implementation; merchants included Mark Twain Bank; went bankrupt 1998
  • Bitcoin (2009): Pseudonymous (not truly anonymous) e-cash using blockchain to prevent double-spending without a central bank
  • Monero/Zcash: True anonymous cryptocurrencies using ring signatures/ZKP
🪙
Micro-payments
Very small online transactions (fractions of a cent to a few cents) where traditional payment systems are impractical due to transaction fees exceeding the payment amount.
  • Problem: Credit card processing costs $0.20-0.30 per transaction — impractical for $0.01 payments
  • Use cases: Pay-per-article news, pay-per-second streaming, IoT machine-to-machine payments, API calls
  • PayWord scheme (Rivest & Shamir): Hash chain — user commits to value; reveals hash chain links as payment; no online CA needed per transaction
  • Lightning Network: Bitcoin payment channels; millions of micropayments settled off-chain; final settlement on blockchain
  • State channels: Both parties lock funds in smart contract; exchange signed state updates off-chain; final state submitted to blockchain
  • Stripe/PayPal fees: 2.9%+$0.30 per transaction — makes sub-$1 payments uneconomical
TOPIC 5 · ENTERPRISE SECURITY — .NET & J2EE CASE STUDIES
🔷
.NET Security Architecture
Microsoft's .NET Framework provides comprehensive security infrastructure for building secure enterprise applications, including authentication, authorisation, cryptography, and code safety.
  • CAS (Code Access Security): Restricts what code can do based on where it comes from (internet zone, local, trusted site) — permissions like file access, network access
  • Evidence: Information about code origin (URL, strong name, zone) used to grant permissions
  • Role-Based Security: System.Security.Principal; check current user's roles: Thread.CurrentPrincipal.IsInRole("Admin")
  • IPrincipal / IIdentity: Standard interfaces for user identity and roles
  • Windows Identity: WindowsPrincipal/WindowsIdentity — uses Windows Kerberos/NTLM
  • Custom Identity: Forms authentication with custom principal (web apps)
  • ASP.NET Authentication: Windows, Forms, Passport (deprecated), Federation (SAML/OAuth)
  • ASP.NET Authorization: [Authorize] attribute, URL authorization, PrincipalPermission
  • Cryptography: System.Security.Cryptography — AesManaged, RSACryptoServiceProvider, SHA256Managed
  • SecureString: Encrypted string in memory — prevents password exposure in memory dumps
☕
J2EE Security Architecture
Java EE (Enterprise Edition) security model for building secure enterprise applications. Supports both declarative security (configuration-based) and programmatic security (code-based).
  • JAAS (Java Authentication and Authorization Service): Pluggable authentication framework; supports Kerberos, LDAP, database, certificate auth
  • LoginModule: Pluggable JAAS component implementing specific auth mechanism
  • Subject: Represents authenticated user; has Set<Principal> (identities) and Set<Credential>
  • LoginContext: Entry point for authentication; calls configured LoginModules
  • JSSE (Java Secure Socket Extension): SSL/TLS implementation; all Java HTTPS uses JSSE; SSLContext, SSLSocket, TrustManager
  • JCE (Java Cryptography Extension): Cipher (AES/DES/RSA), KeyGenerator, Mac, KeyAgreement
  • JGSS (Generic Security Services): GSS-API binding; Kerberos integration; used in enterprise SSO
  • Declarative security: Defined in web.xml / ejb-jar.xml deployment descriptors — no code change needed
  • Programmatic security: HttpServletRequest.isUserInRole("admin"), EJBContext.isCallerInRole()
  • Security realms: File realm, LDAP realm, JDBC realm, Certificate realm
🏢
.NET vs J2EE Security — Comparison
Both platforms provide enterprise-grade security but with different philosophies and APIs.
  • Authentication — .NET: Windows Auth (Kerberos), Forms Auth, Identity Framework, Azure AD / Microsoft Identity Platform
  • Authentication — J2EE: JAAS (pluggable modules), LDAP, Kerberos, DB-based, OAuth2 via Spring Security
  • Authorization — .NET: Role-based via [Authorize] attributes, Claims-based identity, Policy-based
  • Authorization — J2EE: Role-based (declarative in deployment descriptor + programmatic)
  • Crypto — .NET: System.Security.Cryptography — managed + CNG (CryptoNG)
  • Crypto — J2EE: JCA + JCE; provider-based; BouncyCastle as alternative provider
  • SSL/TLS — .NET: SslStream using Windows SChannel; TLS 1.3 supported
  • SSL/TLS — J2EE: JSSE using Java TLS implementation
  • Strength .NET: Deep Windows/AD integration, rich tooling
  • Strength J2EE: Platform-independent, multiple cloud providers, Spring ecosystem
TOPIC 6 · WEB SECURITY PROTOCOLS — HTTPS, S/MIME, PGP
🌐
HTTPS — HTTP over TLS
HTTP Secure = HTTP protocol running over TLS encrypted channel. Port 443. Ensures confidentiality, integrity, and authentication of web communications.
  • How: TLS handshake first → establishes encrypted channel → HTTP requests sent inside
  • HSTS (HTTP Strict Transport Security): Server instructs browser to only use HTTPS for the domain; prevents SSL stripping
  • Certificate pinning: App/browser rejects certificates not matching expected pin; prevents CA compromise attack
  • Mixed content: HTTPS page loading HTTP resources — browsers block or warn; defeats security
  • Let's Encrypt: Free, automated CA; uses ACME protocol; issued 3+ billion certificates
  • HTTP/2 and HTTP/3: Require HTTPS in all major browsers
  • Market: 95%+ of Google Chrome page loads use HTTPS (up from 60% in 2016)
📧
S/MIME — Secure Email
Secure/Multipurpose Internet Mail Extensions. Standard for public key encryption and signing of email messages. Uses X.509 certificates.
  • Provides: Confidentiality (encrypt), Integrity, Authentication, Non-repudiation
  • Signing: Sender signs email with private key; recipient verifies with sender's public key (from cert)
  • Encryption: Sender encrypts with recipient's public key; only recipient can decrypt
  • Certificate needed: Both sender and recipient need X.509 certs from trusted CA
  • MIME wrapping: Uses multipart/signed (clear-signed) or application/pkcs7-mime (opaque)
  • Client support: Outlook, Apple Mail, Thunderbird (with Enigmail)
  • Enterprise use: Digitally signing official documents, financial institutions, legal firms
🔑
PGP — Pretty Good Privacy
Encryption program for email and files. Created by Phil Zimmermann in 1991. Uses Web of Trust instead of CA hierarchy. Now OpenPGP standard (RFC 4880).
  • Web of Trust: No central CA — users sign each other's keys; trust is distributed
  • Key server: Public keys uploaded to keyserver (keys.openpgp.org) for discovery
  • Process: Generate RSA/ECC keypair → upload public key → recipient fetches key → encrypt/sign
  • PGP signature: Sign email with private key → proves authenticity + non-repudiation
  • Hybrid encryption: Generates random AES session key → encrypts message with AES → encrypts session key with recipient's RSA public key
  • GnuPG (GPG): Free OpenPGP implementation; used in Linux package signing, developer workflows
  • S/MIME vs PGP: S/MIME uses CA hierarchy (corporate); PGP uses Web of Trust (individual/dev)
